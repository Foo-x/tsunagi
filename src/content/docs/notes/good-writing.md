## 意識

- 文章で表現できることには限界がある
    - 感情・景色・味などは、どんなに詳細に記述しても、実際の体験には劣る
    - 動画・写真・画像のほうがわかりやすいことも多い
- 文法的な正確さに固執する必要はない
- 文章は、書いてからしばらく経っても読まれるものであり、いろんな人に読まれるものでもある
    - 新語・流行語を使った文章は、時間が経つにつれて痛々しくなる
    - 難しいことばや専門用語を使った文章は、そのことばを知らない人が理解できない
- 流し読みされる前提で書く
    - 特にWebページは2～3割しか読まれない
        - [How Little Do Users Read?](https://www.nngroup.com/articles/how-little-do-users-read/)
    - 以下の各セクションに述べる方法で重要な情報を先に書いたり、無駄な情報を削ったり、わかりやすい構造にしたりする
- 読み手の目的、知識レベル、立場によって文章の構造や必要な情報が変わる
    - 目的
        - たとえば一連の業務の流れを知りたいのか、よくあるトラブルの解決法を知りたいのかなど
    - 知識レベル
        - たとえば用語の説明を加えたり、背景から説明したりするなど
        - 扱う知識によって変わるので注意
            - たとえば中途採用者は業界に関する知識レベルは高いが業務に関する知識レベルは低いなど
    - 立場
        - たとえば管理者向けの手順なのか、一般人向けの手順なのかなど

## 構造

### 文章

- ビジネスで使用する文章は大きく3種類
    - 説明型
        - マニュアルや仕様書など
    - 報告型
        - 報告書や論文など
    - 説得型
        - 企画書や提案書など
- よくある構造
    - PREP（汎用）
        1. 結論
        2. 理由、根拠
        3. 具体例
        4. 結論
    - SDS（汎用）
        1. 要約
        2. 詳細
        3. 要約
    - DESC（説得型）
        1. 客観的な現状、背景（導入）
        2. 主観的な意見、問題点
        3. 解決法の提案
        4. 結論
    - QUEST（説得型）
        1. 問題
        2. 共感
        3. 解決法
        4. 解決後の未来、事例、第三者の意見
        5. 連絡先、ボタン、フォーム、リンク、今後の流れ
    - PASONA（説得型）
        1. 問題
        2. 問題の根拠、共感
        3. 解決法
        4. 期限、人数制限などの絞り込み
        5. 連絡先、ボタン、フォーム、リンク、今後の流れ
- 同じ階層はMECEにする
    - Mutually Exclusive and Collectively Exhaustive: 互いに重複せず、全体で漏れがない
- 一つの段落で扱う話題を一つにする
- 見出しをつける
    - 例
    ```
    AAAAAAAAAAが問題で、その原因はBBBBBBBBBBでした。対策としてCCCCCCCCCCを行います。
    ```
    ↓
    ```
    問題
    AAAAAAAAAA

    原因
    BBBBBBBBBB

    対策
    CCCCCCCCCC
    ```
- ラベルをつける
    - 例
    ```
    AAAAAAAAAAが問題で、その原因はBBBBBBBBBBでした。対策としてCCCCCCCCCCを行います。
    ```
    ↓
    ```
    問題: AAAAAAAAAA
    原因: BBBBBBBBBB
    対策: CCCCCCCCCC
    ```
- 表にする
    - 例
    ```
    AAAAAAAAAAが問題で、その原因はBBBBBBBBBBでした。対策としてCCCCCCCCCCを行います。
    ```
    ↓
    ```
    | 項目 | 内容 |
    | --- | --- |
    | 問題 | AAAAAAAAAA |
    | 原因 | BBBBBBBBBB |
    | 対策 | CCCCCCCCCC |
    ```
- 見出しを文章の流れに含めない
    - 引用しにくいため

### 文

- 順序を説明するときは番号付きの箇条書きを使う
- 並列的な要素を列挙するときは番号なしの箇条書きを使う
    - ただし多用すると逆に読みにくくなるので注意
    - 各要素が単純なものであれば読点や中黒で並べる
- 同じ箇条書きのブロック内では体言止めと用言止めを統一する
- 主語と述語を近くに置く
- 修飾語と被修飾語を近くに置く
- 否定文より肯定文を使う
    - シンプルになるだけでなく、どの語句に否定がかかっているか誤解しにくくなる
    - ただし禁止するときは否定文のほうが表現が強くなるので良い
- 二重否定を使わない
- 受動態より能動態を使う
    - ただし読み手の視点で受動態になるときは受動態を使う
- 一つの文に含まれる意味を一つにする
- 「のように」を否定文で使わない
    - 「AはBのようにCではない」は、「AはBと同様にCではない」「AはBとは違ってCだ」「AはBほどCではない」の3つの解釈ができるため
- andとorが混在する場合は、グループ内を助詞、グループ同士を接続詞と読点でつなげる
    - 「AかBとC」→「AかB、そしてC」「A、もしくはBとC」

#### 順序

- 節を句より前に置く
    - 例
        - 「小さい、人の少ない村」→「人の少ない小さい村」
- 長い修飾語を短い修飾語の前に置く
    - 例
        - 「持ち手の黒い、1年前にコンビニで買った傘」→「1年前にコンビニで買った持ち手の黒い傘」
- 大状況を小状況の前に置く
    - 例
        - 「江戸時代が1603年に始まった」→「1603年に江戸時代が始まった」
        - 「傘も刺さずに1人の男が雨の中を歩いていた」→「雨の中を1人の男が傘も刺さずに歩いていた」
    - 基本的には、いつ、どこで、誰が、何を、なぜ、どうした、の順
    - ルールとしては弱く、強調のために入れ替わりやすい
        - 特に目的（なぜ）は先に置かれることも多い
- 親和度の強弱によって誤読しない順にする
    - 例
        - 「初夏のみどりがもえる夕日に照り映えた」は「みどり」と「もえる」の親和度が強すぎて、パッと見たときに「みどりがもえる夕日」となり不自然
            - 「もえる夕日に初夏のみどりが照り映えた」に変えることで誤読がなくなる
            - 「初夏のみどりが燃える夕日に照り映えた」と漢字に変えても良い
        - 「白い木の実に80%が」は「木」と「実」の親和度が強すぎて「木の実」に見える
            - 「実に80%の白い木が」に変える
            - 「白い木のじつに80%が」とひらがなに変えても良い
        - 「この先生きのこるには」は「先」と「生きのこる」の「生」の親和度が強く、さらに「きのこ」も見えるので誤読しやすい
            - この例は順序で解決できないので、「この先も生きのこるには」と助詞を入れたり、「これからも生きのこるには」のように語を変えたりする

### 要約

- 導入や締めがある場合は削る
- 主語と述語以外で、無くてもわかる情報を削る
    - 冗長な表現（「リズム」セクション参照）
    - 婉曲な表現
    - 具体的な情報
    - 例・比喩
    - 余談
- 無くてもわかる主語を削る

### パラグラフライティング

論理的な文章を効率よく書く方法としてパラグラフライティングがある。

1. 言いたいことを決める
    - それぞれがパラグラフ（≒段落）の先頭の文になる
    - 中心文、主題文、トピックセンテンスなどという
2. 言いたいことを説明する
    - 理由、詳細、言い換え、具体例など
    - 対応する中心文と同じパラグラフに入れる
    - 支持文、サポートセンテンス、サポーティングセンテンスなどという
3. 必要に応じてパラグラフの結論を加える
    - 言いたいことを再度強調したり、パラグラフをまとめたりなど
    - なくても良い
    - 結論文、コンクルーディングセンテンスなどと言う

以下のルールがある。

- 1つのパラグラフでは1つのトピックのみを扱う
    - したがって中心文も1つ
- 中心文だけを抜き出しても文章がつながるようにする
- 接続詞は必要最小限にする

### 英語

- 箇条書きするとき、各項目が独立した文章の場合はそれぞれの最初の文字を大文字にしてピリオドを付ける
    - そうでない場合は小文字にしてピリオドを付けない
- 見出しを大文字にするルールは一般的に2つある
    - センテンスケース (sentence case)
        - 文の最初の文字だけ大文字にする
    - タイトルケース (title case)
        - 文の最初の文字は常に大文字にする
        - 「3文字以下の冠詞、接続詞、前置詞」以外の単語の最初の文字も大文字にする

## 修飾

- 形容詞・副詞は最小限にする
    - 重ねすぎると逆に意味がぼやけるため
    - そもそも対象を完璧に表現することは不可能
    - 論理的な文章では特に
- わかりにくいものをわかりやすくするためだけに比喩を使う
    - もともとわかりやすいものを例える必要はない
    - 比喩によって逆にわかりにくくなることもある
- 修辞技法を使いすぎない
    - 比喩
    - 擬人法
    - 倒置法
    - 体言止め
    - 反語
    - オノマトペ
    - etc

## 句読点

- 2つ以上の文をつなげるときは間に読点を置く
- 単語や句を並列するときはそれぞれの間に読点を置く
    - 1つの単語かつ順序を入れ替えても問題ないときは中黒も使える
- 強調などであえて語順を変えるときは間に読点を置く

## 記号

- 括弧はなるべく使わない
    - 文が構造化されてわかりにくくなるため
    - どうしても使いたいときは注釈にする
        - ※や[1]などとして、文の下や文章の最後に入れる
- 4桁以上の数字には3桁ごとにカンマを入れる

### 英語

- `&` は正式名の一部のときのみ使う
    - 文字数が限られていて `and` が使えないときは例外
- `...` の前後には空白を1つずつ入れる
- `.` の後には空白を1つだけ入れる
- 数字と単位の間には空白を1つだけ入れる

## 漢字とカナ

- ひらがな・カタカナ・漢字を連続させない
    - 単語の切れ目がわかりにくくなるため
    - 漢字をひらがなにしたり、別のことばを使ったりする

## 助詞

- 逆接以外で「が」を使わない
    - 文が長くなりやすいため
    - 逆接として使っているのかわかりにくいため

## リズム

- ひとつの文は40字程度にする
    - 長くても60字まで
- 特別な意図がないときに同じことばを繰り返し使わない
    - 文末
    - 助詞
    - 接続詞
    - 単語
- 節・句・語の長さを揃える
- 冗長な表現を削る
    - 逆に加えることでリズムが良くなったり、削るとニュアンスが変わったりすることもあるので注意
    - 例
        - 「という」
        - 「こと」
        - 「のような」
        - 「基本的に」「一般的に」
        - 「と思う」
        - 「かもしれない」
        - 修飾語
        - 逆接以外の接続詞
        - 本筋と関係のない情報
        - 予防線
        - 同じ意味の繰り返し
        - 指示語 → 具体的な語句にするか削る
        - 処理を実行する → 処理する
        - 増加する → 増える
        - 決定する → 決める
        - 確認を行う → 確認する → 確かめる

### テクニック

多用はNG

- 淡々とした印象や子供っぽい印象を与えたいときは、文を短くして文末を揃える
- 緊張感や勢いを表現したいときは文を長くする
    - 句点の代わりに読点を使って文をつなげるのも有効
- 韻を踏む
    - 頭韻・脚韻

## ことば

いつ誰が読んでも違和感がなく、わかりやすいことばを使うのが基本。

- 以下のことばを使う
    - 広く使われている
    - 昔から使われている
        - 対応することばが無ければ新しいことばでもOK
- 造語は使わない
    - 新しい概念を提唱したり、分かりやすさよりもインパクトを重視したりするときは例外
- 紋切り型の表現を避ける
    - 慣用句、ことわざ、安直な比喩のように、便利だが独自性のない表現のこと
    - 手を抜いている印象になるため
    - 例
        - 目から鱗が落ちる
        - 頭を抱える
        - 肩を落とす
        - ゴクゴクと水を飲む
        - ポカポカとした陽気
        - 嬉しい悲鳴
        - 小鹿のような脚
- 感想を書くのではなく、描写する
    - 感想は書き手が感じたことなので読み手は共感できない
- ありきたりなことを書かない
- 表記ゆれを避ける
    - 送り仮名
    - ひらがな・カタカナ・漢字・ローマ字
    - 外来語
        - ヴァイオリンとバイオリン
        - スパゲティとスパゲッティ
    - 末尾長音
        - ユーザとユーザー
        - サーバとサーバー
    - 算用数字と漢数字
        - 数字に意味がある場合は算用数字、そうでない場合や熟語・慣用句では漢数字
        - 「ひとり」「ひとつ」はひらがなにすることも多い
    - 全角・半角
- 専門用語はその説明も含める
    - 想定読者が用語を知っている人しかいない場合は省いても良い
- 正式名を使う
    - スペースの有無や大文字と小文字の違いなど
    - 略語はなるべく避ける
        - 略語を知らない人が理解できず、知っている人でも脳内で変換するコストがかかるため
        - どうしても使いたいときは最初に使う際に正式名を併記するか、略語を定義している箇所にリンクする
        - `k (kilo)` や `M (Mega)` などの大きい数字の略語も文字数が限られているときを除いて避ける
- 重言を避ける
    - 頭痛が痛い、馬から落馬など
- 抽象的な表現は具体的な名詞や数字にする
    - 形容詞は抽象的になりやすい
    - 抽象的な概念を無理に具体化する必要はない
- 引用は最小限にする
    - 自分のことばを使う
- 必要のないことばを使わない
    - 伏線になっているのか、何か特別な意味があるのか、と読者に思わせてしまうため
    - 「チェーホフの銃」や「オッカムの剃刀」というルール
- 指していることばがわかりにくい指示語は避ける
    - 同じことばを使うのが一度だけで済むように順序を工夫する
    - 別の表現で言い換える
    - 繰り返しになるが、同じことばをそのまま使う
- 意味があるのであれば上記のルールに反しても良い
    - 逆に言うと、ルールに反したことばを使うのであれば意味を持たせなければいけない
    - たとえば紋切り型の表現をあえて使って、その表現につっこんだり、少し変化させたりするなどして展開させる手法がある

### 英語

- 短縮形を優先する
    - `can't` や `it's` など
    - 短縮しないと堅苦しいため
- 特定の人物を指す場合を除いて、性別に依存しない名詞を使う
    - `they` や `a member` など
- zeroからnineまでは数字を使わない
    - 文字数が限られているとき、見出しの先頭、パーセント、吹き出し内、統計などは例外
        - パーセントは数字と `%` を使う

### 論理的な文章

論文やビジネス文書などの論理的な文章に使うことばのルール。

- 事実と意見を区別する
    - 事実は客観的で常に正しいが、意見は主観的で正しいとは限らないため
    - 事実の例
        - このビルは築30年だ
        - このビルは隣のビルより古い
    - 意見の例
        - このビルは古い
    - 事実には証拠を示すことで正しさを証明する
        - 証拠を示せないのであれば意見として書かなければならない
    - 意見には根拠を示すことで説得力を持たせる
        - 根拠には事実が使われる
    - 「と思う」や「と考える」を文末に付けると意見であることがわかりやすくなる
- 感情・ニュアンスを含めない
    - 無駄な情報のため
    - 「してしまった」→「した」
- 話しことばを使わない
    - 方言
    - 音便
        - 「すいません」→「すみません」
        - 「したんですが」→「したのですが」
    - 省略
        - 「してる」→「している」
        - 「業務開始する」→「業務を開始する」
        - 「会社行く」→「会社に行く」
        - 「スマホ」→「スマートフォン」
        - 「携帯」→「携帯電話」
        - パソコンのように略さないと不自然なものはそのままで良い
    - その他
        - 「っていう」→「という」
        - 「AだしB」→「AでB」
        - 「していて」→「しており」
        - 「しなくて」→「せずに」
        - 「じゃない」→「ではない」
        - 「だから」→「したがって」「そのため」
        - 「でも」「だけど」→「しかし」「だが」
        - 「とても」「すごく」→「非常に」「極めて」
        - 「もっと」→「より」「さらに」
        - 「ちょっと」→「多少」
        - 「だいたい」→「およそ」「約」「ほぼ」
        - 「やっぱり」→「やはり」
- 必要以上に丁寧にしない
    - たとえば取扱説明書で「〜してくださいますようお願い申し上げます」のような敬意表現や「お手数をおかけしますが」のようなクッション言葉は不要
        - クッション言葉は除き、「〜してください」で良い

## 見栄え

### レイアウト

- 行間は0.5文字から1文字ほど空ける
- 1行は30文字から40文字ほどにする
- 左揃えにする
    - 中央揃えはガタガタな印象になる
    - 表紙のように特殊なブロックで、その中だけで中央揃えにするのはOK
        - その場合はそのブロックをすべて中央揃えにする
    - 中央揃えの文章に枠をつけて、その枠の大きさを同じにすることで揃っているように見せるテクニックもある
- 単語や句の途中で折り返さない
- 文の折り返しによって1行が3文字以下にならないようにする
    - 文の長さを調節するか、新しい文を加える

#### Web

見る端末によって幅が変わるので、通常の文章とはルールが少し変わる。

- 単語や句の途中で折り返してもOK
    - CSSの`word-wrap: break-word;`を使って、なるべく折り返さないようにはする
- 文の折り返しによって1行が3文字以下になってもOK
- カジュアルな文章では、PCで見たときの2~3行ごとに空行が入るようにして、1文ごとに改行を入れるようにする
    - 集中して読まなくても意味を理解できるようにするため
    - スマホから見るとPCより行数が多くなるので特に注意
    - 1行が短くなってしまう場合は改行せずに文を続けてもOK

### 配色・装飾

- 色は使いすぎない
    - 「無彩色」「メインの有彩色」「強調の有彩色」の3種類のみ
    - メインの有彩色と強調の有彩色は、暖色系と寒色系から1つずつ選ぶか、明度に差をつける
        - 明度が近く、かつ暖色系同士・寒色系同士だと色弱者が区別できないため
    - 有彩色は、色が持つ印象に注意する
        - 青い文字で「熱い」と書くと混乱する
    - 彩度・明度は低めにする
        - 高いと単純に見にくい
        - ディスプレイやプロジェクタによっては見えなくなる可能性もある
    - 背景の色は、印刷を前提にするなら白、投影を前提にするなら白or黒
        - 投影する場所が白いのであれば、背景の色は黒いほうがやや見やすい
        - 背景の色を黒にするのであれば、有彩色の明度をやや高くする
    - 文字の色は、背景の色が白なら黒、黒なら白
- 装飾は使いすぎない
    - 煩雑に見える
- 強調するときは、まず太字を検討する
    - 強調のために色が頻繁に変わると煩雑に見える
- 再利用されるドキュメントはなるべく装飾しない
    - コピペしにくくなるため

### フォント

- 文字を小さくしすぎない
    - Web上であれば16px以上
- 細い文字は使わない
    - 視力の悪い人が見えない
        - 文字を大きくすると改善できる
    - 背景色によってはつぶれて見える
        - 文字を大きくしても改善できない
    - 現代的・都会的・女性的な印象を与えられるので、適切な場所で使うのはもちろんOK

### 画像・映像

- 大きさをそろえる
    - 特に横幅
- ノイズを含めない（チェーホフの銃）
    - 例
        - 服のロゴやテキスト
        - 背景に映り込んだ家具・本・人物
    - そもそも映さないようにするか、画像編集で消したりモザイクをかけたりする
